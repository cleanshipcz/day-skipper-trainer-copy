## Estimation Methodology

All estimates use **dev-days** (1 dev-day = ~6 productive hours of focused development).

**How estimates were derived:**
- Each user story is estimated individually based on: number of new components, data files, interactive tools, and integration points.
- Epic totals are the sum of their story estimates.
- Phase totals are the sum of their epic totals.
- Ranges reflect uncertainty in interactive tool complexity (lower bound = reuse existing patterns; upper bound = novel SVG/canvas work).

**Reconciliation with `docs/FEATURES.md`:**
- `FEATURES.md` uses top-down T-shirt sizes (Small/Medium/Large) with indicative day ranges for product-level prioritisation.
- This document uses bottom-up story-level estimates for engineering scheduling.
- Where the two diverge, **this document is authoritative** for sprint planning and delivery tracking.

## Global Definition of Done

Every task is **Done** when ALL of the following are true:

1. **Code complete** — implementation merged to feature branch, TypeScript strict-mode clean.
2. **Unit tests** — all new public functions/components have unit tests; coverage >= 90% on new code (see [Coverage Tooling](#coverage-tooling) below).
3. **Integration tests** — component interactions and data flows tested.
4. **No regressions** — `npm run test -- --run`, `npm run typecheck`, and `npm run lint` all pass.
5. **Responsive** — renders correctly at 375px, 768px, and 1280px viewports.
6. **Progress tracking** — topic completion persists to Supabase via existing `useProgress` / `progressPersistence` infrastructure.
7. **Points awarded** — points granted on meaningful completions (theory read, quiz passed, tool used).
8. **Accessible** — interactive elements keyboard-navigable; images have alt text.
9. **Documentation** — `docs/FEATURES.md` coverage map updated to reflect new status.
10. **No secrets** — no API keys, tokens, or credentials in committed code.
11. **Database migrations** — any new Supabase tables include a migration file following [Migration Conventions](#supabase-migration-conventions) below.
12. **RLS policies** — every new table containing user data has Row Level Security enabled with appropriate policies (see [RLS Requirements](#rls-policy-requirements) below).

### Supabase Migration Conventions

**Tooling:** Migrations are managed via the [Supabase CLI](https://supabase.com/docs/guides/cli).

**Creating a new migration:**
```bash
npx supabase migration new <descriptive_name>
```
This generates a timestamped file in `supabase/migrations/` (e.g., `20260401120000_add_exam_results_table.sql`).

**Naming convention:**
- Format: `<YYYYMMDDHHMMSS>_<snake_case_description>.sql`
- The timestamp prefix is auto-generated by the CLI.
- The description suffix must be human-readable (e.g., `add_question_reviews_table`, `add_rls_policies_for_badges`).
- Existing migrations use both UUID-based names (legacy, from the initial Supabase project setup) and descriptive names. **New migrations must use descriptive names only.**

**Existing migrations (for reference):**
| File | Purpose |
|------|---------|
| `20251201152907_*.sql` | Add `points` column, `answers_history`, indexes |
| `20251201175517_*.sql` | Add unique constraint on `user_progress` |
| `20260209105000_increment_user_points_rpc.sql` | `increment_user_points` RPC function |

**Applying migrations locally:**
```bash
npx supabase db reset   # Recreates local DB from all migrations
npx supabase db push     # Applies pending migrations to linked project
```

**Rollback expectations:**
- Supabase CLI does not support automatic rollback. Each migration must be **forward-only**.
- If a migration needs to be reverted, create a new migration that reverses the changes (e.g., `DROP TABLE`, `ALTER TABLE DROP COLUMN`).
- Before applying to production, test migrations locally with `npx supabase db reset` to verify the full migration chain applies cleanly.
- **Never modify an already-applied migration file.** Always create a new one.

**One migration per feature boundary:**
- Each story that introduces schema changes should create exactly one migration file.
- Combine related DDL (table creation + indexes + RLS policies) into a single migration when they are part of the same story.

### RLS Policy Requirements

Every new Supabase table that stores user-specific data **must** have Row Level Security (RLS) enabled. This is enforced as DoD item #12.

**Pattern (derived from existing `user_progress` and `profiles` tables):**

```sql
-- 1. Enable RLS on the table
ALTER TABLE <table_name> ENABLE ROW LEVEL SECURITY;

-- 2. Allow authenticated users to manage only their own rows
CREATE POLICY "Users manage own <table_name>"
  ON <table_name> FOR ALL
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);
```

**When to use split policies (SELECT vs INSERT vs UPDATE):**
- Use a single `FOR ALL` policy when users need full CRUD on their own rows (e.g., `question_reviews`, `daily_activity`).
- Use separate `FOR SELECT` / `FOR INSERT` policies when users should not be able to delete or update rows (e.g., `user_badges` — once earned, badges are immutable).

**Tables requiring RLS (from `docs/FEATURES.md` Appendix):**

| Table | Policy Type | Rationale |
|-------|------------|-----------|
| `question_reviews` | `FOR ALL` | Users manage their own review schedule |
| `exam_results` | `FOR ALL` | Users manage their own exam history |
| `user_badges` | `SELECT` + `INSERT` only | Badges are immutable once earned; no user-initiated DELETE/UPDATE |
| `daily_activity` | `FOR ALL` | Users manage their own activity log |

**Verification:** After migration, confirm RLS is active by querying `pg_tables` or attempting an unauthenticated select — it should return zero rows.
